Hiroshi is an always-on personal background AI assistant running entirely on your local machine. Built from the ground up to utilize local hardware resources efficiently, it provides a stable environment for autonomous workflows without relying on external cloud APIs or remote processing nodes.

---

## 🔒 Private by Default Architecture

All operations performed by Hiroshi are hosted and executed within your local hardware sandbox.
*   **Zero External Tracking**: Your code files, chat logs, and vector database remain on your local drive.
*   **Offline-First LLM Connections**: Hiroshi talks to local model deployments (like Ollama) hosted on your loopback interfaces.
*   **Physical Sandbox Isolation**: File modifications are restricted to a dedicated subfolder on your machine, preventing accidental data loss outside whitelisted areas.

---

## ⚡ System Footprint

Because Hiroshi is built as a compiled background executable asset rather than a resource-heavy interpreter app:
*   **Idle Memory Footprint**: Consumes **less than 40 MB of RAM** while waiting for tasks in the background.
*   **Routing Overhead**: Swaps instructions and processes internal loops in **under 5 milliseconds**, delivering instantaneous response cycles.

## 📂 Complete Home Folder Structure

Upon its initial boot sequence, Hiroshi creates a secure configuration and storage directory at `~/.hiroshi/` (inside your user home directory).

```text
~/.hiroshi/
├── config.toml         # Active port routes, model choices, and security settings
├── hiroshi.db          # Hardened local SQLite database tracking chats and vector spaces
├── hiroshi.log         # Running logs detailing telemetry, error bounds, and execution speeds
├── AGENTS.md           # The profile blueprints declaring agent roles and allowed tools
├── SOUL.md             # Directives controlling identity, tone constraints, and values
├── USER.md             # Your context profile, project info, and personal stack preferences
├── backups/            # Directory containing weekly snapshot backups of your configurations
├── memory/             # Archive folders containing daily chat logs and Master MEMORY.md
└── skills/             # Dynamic polyglot script tools (Python, Bash, Node.js)
```

### File and Directory Descriptions:

1.  **`config.toml`**: The main configuration manifest. It controls ports, local Ollama API endpoints, Model Context Protocol (MCP) server blocks, and multi-channel credential details (Telegram, Discord, and Slack Socket Mode).
2.  **`hiroshi.db`**: A self-contained SQLite database storing session history, tool logs, and floating-point vector blobs representing the meaning of past conversations.
3.  **`hiroshi.log`**: A text log capturing execution steps, timing metrics, and error stack details. Consult this file if debugging setup errors.
4.  **`AGENTS.md`**: A markdown file declaring the system prompts and tool lists for your agent personas (such as Architect and Developer).
5.  **`SOUL.md`**: Behavioral directives that govern how the assistant communicates. This forces the agents to stick to a consistent, neutral tone.
6.  **`USER.md`**: A developer bio file. Write your coding language preferences, stack choices, and project context here so the agents adapt code templates automatically.
7.  **`backups/`**: Snapshots of your database, configs, and agent profiles captured automatically at scheduled intervals to protect against configuration corruptions.
8.  **`memory/`**: Holds logs representing past conversations:
    *   `YYYY-MM-DD.md`: Raw logs of conversations, organized by day.
    *   `MEMORY.md`: A summarized context profile detailing custom setup instructions, key architectural rules, and project milestones distilled by background loops.
9.  **`skills/`**: The dynamic tools directory. Place customized automation scripts (e.g. Python scraper, compiler task runner) in this folder to register them as tools.
