title: First-Time Setup
description: Run the onboarding setup terminal wizard to initialize your configuration settings.
---

When Hiroshi is launched for the first time without an existing configuration, it detects the missing `config.toml` file and automatically starts an interactive console setup wizard.

---

## 🚀 Setup Walkthrough

The wizard guides you through local hardware discovery and initial security configuration.

### 1. Hardware Discovery Probe
The onboarding engine sweeps active loopback ports (checking `http://127.0.0.1:11434`) to verify if a local **Ollama** server is running.
*   *Tip*: Make sure Ollama is launched and active in your taskbar before starting the onboarding command.

### 2. Model Allocation Selection
Once the local port is found, Hiroshi query-lists all models you have pulled on your local Ollama engine. Use the arrow keys in your terminal to select your active coding model (e.g. `qwen2.5-coder:1.5b`).

### 3. Telegram Gateway Pairing
To let you chat with your coding agents remotely, the terminal prompts for:
*   **Telegram Bot Token**: The API token generated by BotFather (e.g. `123456:ABC...`).
*   **Whitelisted User ID**: Your numerical Telegram account identifier (e.g. `987654321`) retrieved from `userinfobot`. Hiroshi blocks messages from any other user IDs.

---

## ⚙️ The `config.toml` Reference

The wizard saves your answers inside `~/.hiroshi/config.toml`. You can modify this file at any time using a text editor to update your settings.

```toml
# Main settings for the background daemon
[engine]
system_name = "Hiroshi"
log_level = "info" # Logs trace info, warnings, and errors

# Local Ollama LLM provider configurations
[ollama]
host = "http://127.0.0.1:11434"
model = "qwen2.5-coder:1.5b"
temperature = 0.2
context_window = 4096

# Remote security and file system permissions
[security]
sandbox_path = "~/.hiroshi/workspace"
allowed_binaries = ["cargo", "git", "python", "rustfmt"] # Commands agents can run
```

### Key Parameter Settings:

*   **`engine.log_level`**: Controls logging verbosity. Set to `debug` for detailed diagnostic metrics or `info` for standard operations.
*   **`ollama.model`**: Specifies the local LLM model name.
*   **`ollama.temperature`**: Controls response creativity. Lower values (like `0.2`) are recommended for code generation to reduce syntax errors.
*   **`security.sandbox_path`**: Defines the target directory path where your agents are allowed to read, write, or modify code files.
*   **`security.allowed_binaries`**: A whitelist of command binaries that agents are allowed to execute inside the workspace. Any command not listed here will be blocked.
