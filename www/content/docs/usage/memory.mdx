Hiroshi utilizes a dual-tier memory system to record past conversations and reference relevant workspace facts.

---

## 📅 Daily Log Journals

Every conversation you have with your agents is logged. At the end of the day (or when triggered by background schedules), Hiroshi exports the daily chat threads as a Markdown log inside your memory directory:

```text
~/.hiroshi/memory/YYYY-MM-DD.md
```

This file serves as a readable diary of all command outputs, agent comments, and code changes made on that day.

---

## 🧹 Weekly Compaction Routine

To prevent your historical logs from growing too large for the AI to read at once, Hiroshi runs an automated cleanup routine every **Sunday at 3:00 AM**.

1.  **Config Backups**: Generates compressed configuration snapshots inside `~/.hiroshi/backups/`.
2.  **Information Distillation**: Prompts the local model to read all daily logs from the past week, filter out chat noise, and extract core project developments.
3.  **Appends to Master Profile**: Writes the distilled milestones to the long-term manifest file at:
    ```text
    ~/.hiroshi/memory/MEMORY.md
    ```
    This master file is sent to agents at the beginning of each session, ensuring they remember historical context.

---

## 🧠 Semantic Search (RAG)

Hiroshi searches your past notes by "meaning and intent" rather than looking for exact spelling matches.

*   **Vector Database Blobs**: When new notes are saved, Hiroshi requests coordinate points (embeddings) from your local model and saves them as binary data inside `hiroshi.db`.
*   **Rapid Lookups**: When you ask a question, Hiroshi runs a local math comparison algorithm (cosine similarity) to search the database. It finds matching historical context in **under 15 milliseconds**.
*   **Context Safety Window**: Relevant matches are automatically added to the agent system prompt, keeping the session under model context limits.

---

## 🔒 Multi-Channel Session Isolation

When you connect Hiroshi to multiple chat platforms (such as Telegram, Discord, and Slack) simultaneously, all conversation sessions are strictly partitioned:

*   **Isolated Session Histories**: Each incoming source maps to a unique compound identifier (e.g., `discord:123456` or `slack:98765`). Conversation histories and logs are tagged with a `session_id` inside `hiroshi.db` to prevent agents from mixing up details across channels.
*   **Vector Search Scope**: Retrieval-Augmented Generation (RAG) queries are scoped to the active session. Queries run in one Discord server won't pull up private keys or logs from a Telegram bot chat, maintaining strict workspace boundary limits.
